## 🎯 第一層：你的系統架構是什麼？

### 核心定位
**一套針對物聯網邊緣設備的 PUF 認證框架，具備防禦機制**

不只是讀取 PUF 數據，而是建構一個完整的端到端（End-to-End）安全系統。

### 系統三層架構

#### 1️⃣ **用戶介面層 (UI/Frontend)**
- **工具**: Streamlit (app.py)
- **功能**: 
  - 實時認證流程展示
  - 歷史紀錄查詢與篩選
  - 設備狀態監控
  - 量化圖表展示 (ROC 曲線、Hamming Distance 分佈)

#### 2️⃣ **控制與通訊層 (Server/Backend)**
- **核心引擎**: Python + MQTT Broker (Mosquitto)
- **安全邏輯**:
  - VRF (可驗證隨機函數) Challenge 生成
  - HMAC 完整性驗證
  - Timestamp 防重放攻擊
  - SQLite 審計日誌 (auth_history + crp_records)

#### 3️⃣ **邊緣設備層 (Nodes)**
- **實現**: node.py (IoT 模擬器)
- **數據源**: 真實 SRAM PUF 響應 (從 Nature 2023 數據集)
- **職責**:
  - 接收 MQTT Challenge
  - 查表返回 HMAC-signed Response
  - 支持多設備並行認證

### 認證流程時序圖
```
IoT Server (app.py)    │    MQTT Broker    │    Node (node.py)
                       │                   │
生成 VRF Challenge     │                   │
+ Nonce               │                   │
    │                 │                   │
    ├─ 發布 auth/req ─────────────────────>│
    │                 │                   │ 查詢 SRAM PUF Response
    │                 │                   │ 計算 HMAC(SK, Resp+Nonce)
    │                 │    發布 auth/res  │
    │<────────────────────────────────────┤
    │                 │                   │
比對 Hamming Distance  │                   │
驗證 Timestamp & HMAC  │                   │
更新 SQLite 審計日誌   │                   │
    │                 │                   │
```

## 🛡️ 第三層：為什麼你的做法很重要？（核心創新點）

### 問題：傳統 SRAM PUF 的脆弱性

如果攻擊者取得 crps.csv（200MB 的歷史紀錄），他能做什麼？

```
攻擊流程：
1. 用 Scikit-learn（機器學習）訓練一個模型
2. 輸入：Challenge（已知）
3. 輸出：Response（預測）
4. 準確率：95% 以上！

結論：傳統 SRAM PUF 容易被建模破解
```

### 你的解決方案：三層防禦

#### 防禦層 1：VRF (可驗證隨機函數)
- **傳統做法**：直接返回 SRAM Response
- **你的做法**：Response → VRF 處理 → 變成不可預測的 Token
- **效果**：即使攻擊者有歷史數據，也看不出新 Token 的規律

#### 防禦層 2：HMAC 完整性驗證
- **防禦目標**：中間人攻擊 (MITM)、數據篡改
- **實現**：HMAC(Secret Key, Response + Nonce)
- **效果**：任何位元修改都會被檢測

#### 防禦層 3：動態 Timestamp + Anti-Replay
- **防禦目標**：重放攻擊 (Replay Attack)
- **實現**：每次 Challenge 帶上時戳，伺服器驗證新鮮度
- **效果**：同一個 Response 無法被使用第二次
