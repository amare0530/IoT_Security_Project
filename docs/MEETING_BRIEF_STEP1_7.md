# SRAM PUF IoT Authentication Meeting Brief

## 1. One-Line Positioning

This project is not only an encryption demo. It is a low-cost IoT device authentication pipeline: SRAM PUF provides device-specific key material, stable-bit preprocessing improves reliability, Fuzzy Extractor reconstructs a stable key, and HMAC over MQTT verifies device identity.

## 5. Academic And Practical Comparison

| Issue | Traditional Fixed-Key IoT | Our SRAM PUF Pipeline |
| --- | --- | --- |
| Device identity | Stored key in flash/config | Hardware-derived key material |
| Clone resistance | Weak if key is extracted | Stronger, because key is tied to SRAM behavior |
| Noise handling | Usually not addressed | Stable-bit preprocessing + FE |
| Replay defense | Often missing in simple demos | Nonce + timestamp + nonce cache |
| Communication | Plain command/response or fixed HMAC | MQTT challenge-response HMAC |

Careful wording:

> HMAC provides authentication and integrity, not encryption. If payload confidentiality is required, we still need TLS or payload encryption.

> Helper data is public in FE designs, but security depends on PUF entropy, bias, and helper-data leakage. We should not claim "zero leakage" without qualification.

## 6. Teacher/Reviewer Questions To Prepare

**Q1. What is A/B/D grading? Is it a tool standard?**

Answer:

> It is our data-driven reliability classification, not a universal tool standard. The basis is bit stability across repeated SRAM startup measurements. A-grade means highly stable, B-grade means usable with FE correction, and D-grade means unstable/discarded. The exact thresholds are experimental design parameters.

Suggested definition:

| Grade | Suggested Meaning |
| --- | --- |
| A | stability >= 0.99 |
| B | 0.90 <= stability < 0.99 |
| D | stability < 0.90, discarded |

**Q2. What is adaptive stable-bit preprocessing?**

Answer:

> It is Step 2. For each device, we compute per-bit stability and generate a per-device mask. It is adaptive because stable bit positions differ across devices. The goal is to reduce BER before Fuzzy Extractor reconstruction.

**Q3. Is authentication for the machine or the user?**

Answer:

> Current scope is device authentication. We verify that the IoT device is the enrolled hardware. User authentication and authorization are upper-layer access control and can be added later.

**Q4. What are we protecting?**

Answer:

> We protect device identity and message authenticity. The server can verify that a message came from an enrolled device and was not modified or replayed. We are not yet encrypting sensor data contents.

**Q5. Why is Inter-HD not close to 0.5?**

Answer:

> The dataset appears to have SRAM bit bias, so raw device uniqueness is weaker than ideal. This is exactly why we need preprocessing, FE, and HMAC rather than relying on raw PUF bits directly.

**Q6. Does Step 6 really reconstruct key from SRAM live?**

Answer:

> Not yet. Step 6 currently uses a PUF-derived key registry produced by Step 3. This simulates the FE output and validates the full MQTT authentication loop. With physical ESP32 SRAM access, the registry lookup would be replaced by live `Rep(response, helper_data)`.

**Q7. Server stores what exactly?**

Answer:

> Current prototype stores a key registry for integration testing. In a stricter design, server should store UID, helper/mask data, and a verifier or protected key record. We should not claim the current prototype stores only `masks.json`.

**Q8. Is this end-to-end encryption?**

Answer:

> No. It is end-to-end authentication/integrity over MQTT. Confidentiality requires TLS or payload encryption.
